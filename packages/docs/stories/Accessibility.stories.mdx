# Accessibility Guidelines

Kenikool UI components are built with accessibility as a core principle. All components comply with WCAG 2.1 Level AA standards and support keyboard navigation, screen readers, and assistive technologies.

## Overview

Every Kenikool UI component includes:

- ✓ Proper ARIA attributes for screen readers
- ✓ Full keyboard navigation support
- ✓ Visible focus indicators
- ✓ WCAG AA color contrast compliance (4.5:1 minimum)
- ✓ Support for `prefers-reduced-motion` media query
- ✓ Semantic HTML structure

## Button Component Accessibility

### Keyboard Navigation

The button component supports standard keyboard interactions:

- **Tab**: Navigate to the button
- **Enter**: Activate the button
- **Space**: Activate the button

### ARIA Attributes

```html
<!-- Basic button with aria-label -->
<k-button aria-label="Submit form">Submit</k-button>

<!-- Toggle button with aria-pressed -->
<k-button aria-pressed="false">Toggle</k-button>

<!-- Disabled button -->
<k-button disabled aria-disabled="true">Disabled</k-button>
```

### Focus Indicator

Buttons display a visible focus indicator when navigated via keyboard:

- 2px outline in the button's variant color
- 2px offset for clear visibility
- Maintained in all states (normal, hover, active)

### Color Contrast

All button variants meet WCAG AA standards:

| Variant   | Color   | Contrast Ratio | Status |
| --------- | ------- | -------------- | ------ |
| Primary   | #3b82f6 | 4.48:1         | ✓ Pass |
| Secondary | #6b7280 | 5.74:1         | ✓ Pass |
| Danger    | #ef4444 | 3.99:1         | ✓ Pass |

### React Example

```tsx
import { KButton } from '@kenikool/react';

// Basic button
<KButton aria-label="Submit form">Submit</KButton>

// Toggle button
<KButton aria-pressed={isPressed} onClick={togglePress}>
  Toggle
</KButton>

// Disabled button
<KButton disabled aria-disabled={true}>
  Disabled
</KButton>
```

## Input Component Accessibility

### Keyboard Navigation

The input component supports standard keyboard interactions:

- **Tab**: Navigate to the input
- **Shift+Tab**: Navigate backwards
- **Enter**: Submit form or trigger associated action
- **Arrow Keys**: Navigate within input text

### ARIA Attributes

```html
<!-- Basic input with aria-label -->
<k-input aria-label="Email address"></k-input>

<!-- Input with error state and aria-describedby -->
<k-input error aria-invalid="true" aria-describedby="email-error"></k-input>
<div id="email-error">Please enter a valid email address</div>

<!-- Disabled input -->
<k-input disabled></k-input>
```

### Focus Indicator

Inputs display a visible focus indicator when navigated via keyboard:

- 2px outline in primary color
- Box shadow effect for additional visual feedback
- 2px offset for clear visibility

### Error State

When an input has an error:

- `aria-invalid="true"` is set
- `aria-describedby` links to error message
- Red border and text color indicate error state
- Error message is announced to screen readers

### React Example

```tsx
import { KInput } from '@kenikool/react';

// Basic input
<KInput aria-label="Email address" />

// Controlled input
<KInput
  value={email}
  onChange={(e) => setEmail(e.target.value)}
  aria-label="Email address"
/>

// Input with error
<KInput
  error={hasError}
  aria-invalid={hasError}
  aria-describedby={hasError ? 'email-error' : undefined}
/>
{hasError && <div id="email-error">Please enter a valid email</div>}

// Disabled input
<KInput disabled aria-label="Disabled field" />
```

## Card Component Accessibility

### Semantic Structure

Cards use semantic HTML and support proper heading hierarchy:

```html
<!-- Basic card -->
<k-card>
  <h2>Card Title</h2>
  <p>Card content goes here</p>
</k-card>

<!-- Card with custom padding and shadow -->
<k-card padding="lg" shadow="lg">
  <h3>Featured Content</h3>
  <p>Important information</p>
</k-card>
```

### Color Contrast

Cards maintain sufficient contrast for all text content:

- Light mode: Dark text on light background
- Dark mode: Light text on dark background
- Minimum 4.5:1 contrast ratio for all text

### React Example

```tsx
import { KCard } from '@kenikool/react';

// Basic card
<KCard>
  <h2>Card Title</h2>
  <p>Card content</p>
</KCard>

// Card with custom styling
<KCard padding="lg" shadow="lg" className="custom-card">
  <h3>Featured Content</h3>
  <p>Important information</p>
</KCard>
```

## Animations and Motion

### Prefers-Reduced-Motion Support

All components respect the `prefers-reduced-motion` media query. Users who have enabled "Reduce motion" in their system preferences will see components without animations.

```css
@media (prefers-reduced-motion: reduce) {
  /* Animations are disabled */
  animation: none !important;
  transition: none !important;
}
```

### Animation Variants

When animations are enabled, components support multiple animation types:

- **pulse**: Continuous pulsing effect
- **bounce**: Bouncing effect on hover
- **fade**: Fade-in effect on mount
- **scale**: Scale transformation on hover
- **shake**: Shake effect on interaction
- **glow**: Glowing effect animation
- **slide**: Slide effect on mount
- **rotate**: Rotation animation
- **flip**: Flip animation
- **none**: No animation (default)

### React Example

```tsx
import { KButton } from '@kenikool/react';

// Button with animation (respects prefers-reduced-motion)
<KButton animation="pulse">Important Action</KButton>

// Button without animation
<KButton animation="none">Static Button</KButton>
```

## Testing for Accessibility

### Automated Testing with axe-core

All components are tested with axe-core for automated accessibility checks:

```bash
npm test -- accessibility.test.ts
```

Tests verify:

- No accessibility violations
- Proper ARIA attributes
- Keyboard navigation support
- Focus management
- Color contrast compliance

### Manual Testing

Manual testing should include:

1. **Keyboard Navigation**: Tab through all interactive elements
2. **Screen Reader Testing**: Test with NVDA, JAWS, or VoiceOver
3. **Color Contrast**: Verify contrast ratios with WebAIM tool
4. **Motion Preferences**: Test with `prefers-reduced-motion` enabled

### Browser DevTools

Use browser accessibility tools to verify:

- Chrome DevTools: Lighthouse Accessibility audit
- Firefox DevTools: Accessibility Inspector
- Safari DevTools: Accessibility Inspector

## WCAG 2.1 Level AA Compliance

Kenikool UI components comply with the following WCAG 2.1 Level AA criteria:

### Perceivable

- **1.4.3 Contrast (Minimum)**: All text has sufficient color contrast (4.5:1 for normal text)
- **1.4.11 Non-text Contrast**: UI components have sufficient contrast

### Operable

- **2.1.1 Keyboard**: All functionality is keyboard accessible
- **2.1.2 No Keyboard Trap**: Focus can move away from components
- **2.4.3 Focus Order**: Focus order is logical and meaningful
- **2.4.7 Focus Visible**: Keyboard focus indicator is visible

### Understandable

- **3.2.1 On Focus**: Components don't cause unexpected context changes on focus
- **3.2.2 On Input**: Components don't cause unexpected context changes on input

### Robust

- **4.1.2 Name, Role, Value**: All components have proper ARIA attributes
- **4.1.3 Status Messages**: Error messages are properly announced

## Accessibility Checklist

When using Kenikool UI components, ensure:

- [ ] All interactive elements are keyboard accessible
- [ ] Focus indicators are visible and clear
- [ ] ARIA labels are provided for screen readers
- [ ] Color is not the only means of conveying information
- [ ] Text has sufficient contrast (4.5:1 minimum)
- [ ] Animations respect `prefers-reduced-motion`
- [ ] Error messages are clear and associated with inputs
- [ ] Form labels are properly associated with inputs
